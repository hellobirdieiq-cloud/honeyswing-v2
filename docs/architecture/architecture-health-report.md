## 1. Executive summary

**Score: 795/1000.** Identical to the first Phase 2 pass — and that stability
is itself a finding: the intervening GoogleSignIn dependency removal and a
full `prebuild --clean` regeneration of `ios/` moved **zero** axes, confirming
the score is driven by durable structural properties, not day-to-day churn.

The analysis core is in excellent health: domain determinism is *proven*
(byte-identical double-run), capture-pipeline rate-independence is complete
(one decimation constant, ms-thresholds throughout, zero stale fps literals),
the persisted data contract is byte-fresh against the live schema, and the
native bridge survived a clean regeneration with zero drift. The losses
concentrate almost entirely in the *assurance* layers: the test gate covers
the pure math superbly but has four missing analysis-critical suites and zero
UI/glue tests (−52); the static gates see only part of the repo — the entire
lib test suite and the 28-file validation-script toolkit are type-checked by
nothing (−38); phase-detection calibration has literal tunable copies in the
coach-mode debug views and a prose-frozen legacy detector (−40); and the
dependency graph carries five dispatcher cycles plus one misclassified
runtime dependency (−40). The codebase remains easier to evolve than its
tooling can currently prove.

With the fresh-data decision, the risk ordering changes even though the
measurements don't: the historically scariest surface (rows written by past
pipeline versions) is about to be emptied, so the top priorities become the
assurance-layer gaps — gate perimeter, test placement, calibration sourcing,
and boundary codification — with the persisted-row contract re-entering as a
cheap, migration-free thing to define *correctly from day one* of the new
data shape.

## 4. Top 20 architecture findings, ranked by long-term impact

## 6. FINAL ARCHITECTURAL ASSESSMENT

*One week as lead architect, architecture only, ranked by long-term
evolvability impact — re-ranked for the fresh-data reality (no historical
swings to preserve). Structural decisions, not code.*

**#1 — Close the static-analysis perimeter to the whole repository.**
Strict TypeScript is this repo's refactor amplifier, and it currently sees
neither the lib test suite, nor scripts/, nor any unimported file. Every
blind spot is a place where "it compiles" is false comfort — including the
persistence tests that will guard the *new* data shape. With old rows gone,
the codebase is about to change faster, not slower; the perimeter must be
complete before that churn starts. Extending it until every source and test
file is loaded by some typecheck config is the highest multiplier available:
it strengthens every future change in every subsystem at zero design risk.
Findings #1, #2, #3.

**#2 — Reallocate test mass to the risk surface and fix the gate's blind
spots.** Test placement mirrors ease, not regression cost: superb coverage of
pure math beside zero coverage of the capture orchestration, coordinate
mapping, and four analysis-critical modules — and the runner cannot even
discover a `.tsx` test yet. Under fresh data this gap is sharper: every swing
in the new corpus will be produced by exactly the unguarded capture path, so
a capture regression now poisons the *only* data there is. Policy, not
sprint: every documented invariant gets an executable guard; the runner
discovers every test-file class before the first such file exists. Findings
#4, #5, #8, #11.

**#3 — Finish calibration single-sourcing and make the legacy freeze a
decision.** Recalibration against real coaching sessions is the product's
core iteration loop, and `EXTERNAL_ASSUMPTIONS` exists precisely to make it a
single edit. That property fails at both edges today: the coach-mode tuning
screens carry uncommented copies, and the legacy detector's freeze is
enforced by nothing. Fresh data strengthens the case — the freeze existed to
keep old unknown-angle swings scoring identically, and with those rows gone
the freeze itself is up for re-decision. Complete by-reference consumption;
declare, guard, or retire the legacy freeze. Findings #6, #7.

**#4 — Declare the domain packages' public API and codify boundary rules.**
The layering is healthy today — by discipline, not mechanism. No allowlist
for the two known escapes, no declared surface separating "product logic UI
may import" from "pipeline internals," mixed import styles that defeat
auditing, and type-cycles binding the dispatcher to its detectors. Writing
the public API down and making the boundary machine-checkable makes the
repo's single most valuable property — the pure, replayable analysis core —
durable under team growth, and unlocks the extraction the `packages/` layout
has always implied. Findings #9, #10, #12, #16.

**#5 — Define the fresh persisted-swing contract from day one.**
Demoted from #1 by the fresh-data decision — and transformed by it: this is
now an *opportunity*, not a debt. The table will be empty; there will never
be a cheaper moment to make the row contract explicit — field registry with
owners and consumers (deciding `category_scores`' fate before the new corpus
accumulates it), real `analysis_version` read semantics, and a validation
gate before pre-staged signals like `reliability.impact` gain consumers. Do
it as the new data shape stabilizes, not after rows pile up — because the
moment they do, this item starts climbing back toward #1. Findings #13, #14,
#15.
